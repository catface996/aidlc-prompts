# 提示词质量评价体系 (Prompt Quality Evaluation Framework, PQEF)

> 用于评估提示词质量的系统化评分框架，帮助持续优化与 AI 的协作效果。

---

## 评分体系概览

### 五大维度

| 维度 | 英文 | 评分项数 | 满分 | 核心问题 |
|------|------|---------|------|---------|
| **Ⅰ. 清晰性** | Clarity | 4项 | 20分 | AI 能否准确理解意图？ |
| **Ⅱ. 可控性** | Control | 4项 | 20分 | 输出是否符合预期？ |
| **Ⅲ. 任务性** | Taskability | 4项 | 20分 | AI 能否直接执行？ |
| **Ⅳ. 鲁棒性** | Robustness | 3项 | 15分 | 异常情况如何处理？ |
| **Ⅴ. 可扩展性** | Scalability | 5项 | 25分 | 是否易于维护复用？ |
| **总计** | | **20项** | **100分** | |

### 评级标准

| 等级 | 分数范围 | 评价 | 建议 |
|------|---------|------|------|
| **S** | 90-100 | 卓越 | 可作为模板参考 |
| **A** | 80-89 | 优秀 | 小幅优化即可 |
| **B** | 70-79 | 良好 | 有改进空间 |
| **C** | 60-69 | 及格 | 需要较多改进 |
| **D** | 40-59 | 较差 | 需要重写 |
| **F** | 0-39 | 不合格 | 完全重构 |

### 单项评分标准 (1-5分)

| 分数 | 评价 | 标准 |
|------|------|------|
| **5** | 优秀 | 完全满足，可作为范例 |
| **4** | 良好 | 基本满足，有小改进空间 |
| **3** | 一般 | 部分满足，需要补充 |
| **2** | 较差 | 大部分缺失，需要较多改进 |
| **1** | 很差 | 几乎没有，需要重写 |

---

## Ⅰ. 清晰性 (Clarity) - 20分

> 核心问题：AI 能否准确理解你的意图？

### 评分项详解

| # | 评分项 | 满分 | 1分标准 | 5分标准 |
|---|--------|------|---------|---------|
| 1 | **目标清晰度** | 5分 | 意图模糊，需要猜测 | 一句话说清要达成的目标 |
| 2 | **背景充分性** | 5分 | 无背景说明 | 业务背景、技术约束、使用场景清晰 |
| 3 | **概念边界清晰** | 5分 | 术语模糊/自造词/多义 | 关键概念有明确定义，无歧义 |
| 4 | **输入结构化程度** | 5分 | 自然语言一大段 | 结构化输入（表格/JSON/分点） |

### 检查清单

```
清晰性自检：
- [ ] 目标是否用一句话说清？
- [ ] 是否提供了足够的背景信息？
- [ ] 关键术语是否有明确定义？
- [ ] 是否存在"大概"、"可能"、"合适"等模糊词？
- [ ] 输入信息是否结构化？
```

### 示例对比

```markdown
❌ 低分示例 (5/20)：
"帮我写个好用的登录功能"

问题：
- "好用"是模糊词，没有标准
- 未说明技术栈、业务背景
- 无结构化信息

✅ 高分示例 (18/20)：
"## 任务目标
实现用户邮箱+密码登录功能

## 背景
- 项目：电商后台管理系统
- 用户量：约 5000 DAU
- 安全要求：高（涉及订单数据）

## 技术约束
- 技术栈：React + Node.js + PostgreSQL
- 认证方式：JWT token，有效期 24 小时

## 关键定义
- 登录失败：密码错误或账户不存在
- 账户锁定：连续 5 次失败后锁定 15 分钟"
```

## Ⅱ. 可控性 (Control) - 20分

> 核心问题：输出是否符合预期？

### 评分项详解

| # | 评分项 | 满分 | 1分标准 | 5分标准 |
|---|--------|------|---------|---------|
| 5 | **输出格式规范度** | 5分 | 无格式要求 | 有明确模板/schema/示例 |
| 6 | **约束明确性** | 5分 | 无约束 | MUST/NEVER/边界清晰 |
| 7 | **风格一致性** | 5分 | 无风格要求 | 语气/术语/格式风格明确 |
| 8 | **验证与校验机制** | 5分 | 无验证 | 有 Gate Check/验收标准 |

### 约束语言强度表

| 强度 | 关键词 | 适用场景 |
|------|--------|---------|
| **绝对强制** | MUST, SHALL, ALWAYS | 不可违反的规则 |
| **绝对禁止** | NEVER, MUST NOT, STRICTLY PROHIBITED | 危险行为 |
| **强烈建议** | SHOULD, RECOMMENDED | 最佳实践 |
| **可选** | MAY, COULD, OPTIONAL | 锦上添花 |

### 检查清单

```
可控性自检：
- [ ] 是否定义了输出格式（模板/示例）？
- [ ] 是否有 MUST/NEVER 约束？
- [ ] 是否明确了风格要求？
- [ ] 是否有验收/检查清单？
- [ ] 约束是否可验证（非主观判断）？
```

### 示例对比

```markdown
❌ 低分示例 (4/20)：
"写代码时注意安全"

问题：
- "注意安全"太模糊
- 无具体约束
- 无验证标准

✅ 高分示例 (18/20)：
"## 输出格式 (REQUIRED)
返回 JSON 格式：
{
  \"code\": \"string\",
  \"explanation\": \"string\",
  \"test_cases\": [\"string\"]
}

## 安全约束 (NON-NEGOTIABLE)
**MUST**:
- 使用参数化查询
- 对用户输入进行校验

**NEVER**:
- 拼接 SQL 语句
- 在日志中记录密码

## 风格要求
- 代码风格：遵循 ESLint Airbnb
- 注释语言：中文
- 变量命名：camelCase

## Gate Check
- [ ] 代码中无字符串拼接 SQL
- [ ] 用户输入都有校验
- [ ] 通过 ESLint 检查"
```

## Ⅲ. 任务性 (Taskability) - 20分

> 核心问题：AI 能否直接执行？

### 评分项详解

| # | 评分项 | 满分 | 1分标准 | 5分标准 |
|---|--------|------|---------|---------|
| 9 | **步骤明确性** | 5分 | 无步骤指引 | 有编号步骤/流程图 |
| 10 | **动作可执行性** | 5分 | 需多轮澄清 | 读完即可开始执行 |
| 11 | **完整性** | 5分 | 缺少关键信息 | 输入/输出/资源都提供 |
| 12 | **工具规范性** | 5分 | 未说明工具 | 明确使用哪些工具/API |

### 检查清单

```
任务性自检：
- [ ] 是否有明确的执行步骤（1, 2, 3...）？
- [ ] AI 读完后能否立即开始？
- [ ] 输入数据/资源是否提供？
- [ ] 是否说明使用什么工具/API？
- [ ] 遇到分支时是否有决策条件？
```

### 触发词映射示例

```
触发词映射表：
┌─────────────────────────────────┬──────────────────────────────┐
│ 用户输入                         │ 执行动作                      │
├─────────────────────────────────┼──────────────────────────────┤
│ "实现这个功能"                   │ → 分层实现 + 测试            │
│ "设计 API 接口"                  │ → RESTful 设计 + 文档        │
│ "优化性能"                       │ → 瓶颈分析 + 优化方案        │
│ "帮我 Review 代码"               │ → 代码审查 + 安全检查        │
└─────────────────────────────────┴──────────────────────────────┘
```

### 示例对比

```markdown
❌ 低分示例 (5/20)：
"分析一下这个系统的性能问题"

问题：
- 无具体步骤
- 未提供系统信息
- 未说明分析工具

✅ 高分示例 (18/20)：
"## 任务：分析 API 性能瓶颈

## 执行步骤
1. 读取 `src/api/` 目录下的路由文件
2. 识别响应时间 > 500ms 的接口
3. 分析每个慢接口的数据库查询
4. 输出优化建议

## 输入资源
- 代码目录：./src/api/
- 性能日志：./logs/perf.log
- 数据库 schema：./docs/db-schema.md

## 使用工具
- 代码分析：Read + Grep
- 日志分析：Bash (grep/awk)

## 决策条件
- 如果接口无数据库调用 → 检查外部 API 调用
- 如果查询未命中索引 → 建议添加索引"
```

## Ⅳ. 鲁棒性 (Robustness) - 15分

> 核心问题：异常情况如何处理？

### 评分项详解

| # | 评分项 | 满分 | 1分标准 | 5分标准 |
|---|--------|------|---------|---------|
| 13 | **歧义处理机制** | 5分 | 无处理 | 有 [NEEDS CLARIFICATION] 机制 |
| 14 | **错误处理机制** | 5分 | 无处理 | 有错误场景和处理方式 |
| 15 | **任务失败恢复机制** | 5分 | 无处理 | 有回退/重试/降级方案 |

### 检查清单

```
鲁棒性自检：
- [ ] 遇到歧义时如何处理？
- [ ] 错误场景是否列举？
- [ ] 任务失败后如何恢复？
- [ ] 是否有降级/回退方案？
- [ ] 边界条件是否考虑（空值/异常/超限）？
```

### 鲁棒性设计模式

```markdown
## 歧义处理
如果需求存在歧义：
1. 列出 [NEEDS CLARIFICATION] 问题清单
2. 提供可选方案供选择
3. 等待确认后再执行

## 错误处理
| 错误场景 | 处理方式 |
|---------|---------|
| 文件不存在 | 报错并列出可能路径 |
| 权限不足 | 提示所需权限 |
| 格式错误 | 显示期望格式示例 |

## 失败恢复
- **重试**：网络超时重试 3 次
- **降级**：API 不可用时使用缓存
- **回退**：代码修改前备份原文件
```

### 示例对比

```markdown
❌ 低分示例 (3/15)：
"修改数据库配置"

问题：
- 配置错误怎么办？
- 修改失败怎么恢复？
- 没有验证机制

✅ 高分示例 (14/15)：
"## 任务：修改数据库连接配置

## 歧义处理
如果配置文件路径不明确，优先检查：
1. ./config/database.yml
2. ./.env
3. 列出 [NEEDS CLARIFICATION] 请求确认

## 错误处理
| 场景 | 处理 |
|------|------|
| 配置文件不存在 | 报错，不创建新文件 |
| 格式不合法 | 显示当前格式和期望格式 |
| 连接测试失败 | 回滚配置，报告错误详情 |

## 失败恢复
1. 修改前备份原配置
2. 修改后测试连接
3. 测试失败则自动回滚"
```

## Ⅴ. 可扩展性 (Scalability) - 25分

> 核心问题：是否易于维护复用？

### 评分项详解

| # | 评分项 | 满分 | 1分标准 | 5分标准 |
|---|--------|------|---------|---------|
| 16 | **模块化程度** | 5分 | 单一大块 | 可独立修改的模块 |
| 17 | **角色定义** | 5分 | 无角色 | 角色职责边界清晰 |
| 18 | **工作流组合能力** | 5分 | 无法组合 | 可与其他流程串联 |
| 19 | **可维护性** | 5分 | 难以修改 | 更新某部分不影响其他 |
| 20 | **模板化程度** | 5分 | 只适用特定场景 | 可参数化复用 |

### 检查清单

```
可扩展性自检：
- [ ] 是否使用占位符（[变量名]）而非硬编码？
- [ ] 各部分是否可以独立修改？
- [ ] 角色定位是否清晰？
- [ ] 是否可以与其他工作流组合？
- [ ] 是否可以作为模板复用？
```

### 模块化结构示例

```markdown
# [角色名称] 角色定义

## 角色定位
一句话说明角色职责

## 核心原则 (NON-NEGOTIABLE)
- 原则1
- 原则2

## 工作流程
### Phase 0: 上下文加载
### Phase 1: 分析
### Phase 2: 执行

## 输出格式 (REQUIRED)
[模板]

## 与其他角色的协作
- 上游：[角色] 提供 [什么]
- 下游：向 [角色] 交付 [什么]

## 参数化配置
- [技术栈]：React/Vue/Node.js
- [输出格式]：Markdown/JSON/YAML
- [语言]：中文/英文
```

## 评分流程

### Step 1: 快速扫描 (2分钟)

```
快速检查：
- [ ] 有没有明确的目标？（无 → 清晰性 #1 低分）
- [ ] 有没有结构/分段？（无 → 清晰性 #4 低分）
- [ ] 有没有输出格式？（无 → 可控性 #5 低分）
- [ ] 有没有约束？（无 → 可控性 #6 低分）
- [ ] 有没有执行步骤？（无 → 任务性 #9 低分）
```

### Step 2: 逐维度评分 (5分钟)

按照五个维度、20个评分项依次评分。

### Step 3: 计算总分

```
总分 = Ⅰ + Ⅱ + Ⅲ + Ⅳ + Ⅴ

注意：
- Ⅳ. 鲁棒性 满分 15 分（3项×5分）
- 其他维度满分 20 分或 25 分
```

### Step 4: 输出改进建议

优先改进：
1. 得分 ≤2 的评分项
2. 影响任务执行的问题
3. 缺失的关键信息

## 常见扣分项速查

| 问题 | 影响评分项 | 修复方法 |
|------|-----------|---------|
| 目标模糊 | #1 目标清晰度 | 一句话说清目标 |
| 无背景信息 | #2 背景充分性 | 补充业务/技术背景 |
| 术语不清 | #3 概念边界 | 定义关键术语 |
| 一大段文字 | #4 输入结构化 | 使用表格/列表结构化 |
| 无输出格式 | #5 输出格式规范 | 提供模板或示例 |
| 无约束 | #6 约束明确性 | 添加 MUST/NEVER |
| 无执行步骤 | #9 步骤明确性 | 添加编号步骤 |
| 需要澄清 | #10 动作可执行性 | 补充缺失信息 |
| 无错误处理 | #14 错误处理 | 列出错误场景和处理方式 |
| 硬编码值 | #20 模板化程度 | 使用变量占位符 |

## 维度权重说明

```
权重设计理念：

可扩展性 (25%) ─────────────────────────────────■■■■■■■■■■■■■
  └─ 提示词的长期价值在于可复用和可维护

清晰性 (20%) ───────────────────────────────────■■■■■■■■■■
  └─ AI 理解是一切的基础

可控性 (20%) ───────────────────────────────────■■■■■■■■■■
  └─ 输出符合预期是核心目标

任务性 (20%) ───────────────────────────────────■■■■■■■■■■
  └─ 直接可执行减少来回沟通

鲁棒性 (15%) ───────────────────────────────────■■■■■■■■
  └─ 异常处理保证稳定性
```

## 附录：Anthropic 官方评估方法

基于 Anthropic 教程附录 10.3 总结的三种评估方式：

### 1. 代码评估 (Code-Based Grading)

适用于有明确正确答案的任务：

```python
def grade_exercise(text):
    # 检查输出是否包含预期内容
    return "expected_keyword" in text.lower()

# 示例：情感分类评估
def grade_sentiment(output, golden_answer):
    return output.lower() == golden_answer.lower()
```

**适用场景**：
- 分类任务（情感分析、意图识别）
- 格式验证（JSON 格式、特定标签）
- 关键词匹配

### 2. 人工评估 (Human Grading)

适用于创意性、主观性任务：

```markdown
## 评分标准 (Rubric)

一篇高质量的文章应该：
- 有清晰的论点
- 段落结构良好
- 有说服力的例子
- 语言流畅
```

**适用场景**：
- 文章/创意写作
- 开放式问答
- 复杂推理

### 3. 模型评估 (Model-Based Grading)

让 Claude 评估自己的输出：

```python
def build_grader_prompt(output, rubric):
    return f"""Assess the quality based on this rubric:
    <rubric>{rubric}</rubric>
    <output>{output}</output>
    Provide a score from 1-5, where 1 is poor and 5 is excellent."""
```

**适用场景**：
- 摘要质量评估
- 事实核查
- 语气分析

### 评估方式选择指南

| 任务类型 | 推荐评估方式 | 原因 |
|---------|-------------|------|
| 分类/选择 | 代码评估 | 有明确正确答案 |
| 摘要/提取 | 模型评估 | 需要语义理解 |
| 创意写作 | 人工评估 | 主观判断 |
| 事实核查 | 模型评估 | 可定义评分标准 |
| 格式验证 | 代码评估 | 可程序化检查 |
